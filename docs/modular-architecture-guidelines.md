# 웹앱 완전 모듈화를 위한 구조 원칙 문서

이 문서는 "나중에 기능을 많이 붙여도 구조가 무너지지 않게 하기 위한 기준"이다.
단순한 아이디어 메모가 아니라, 실제 구현 중 판단 기준으로 계속 참조하는 운영 문서다.

이 프로젝트의 기본 전제는 다음과 같다.

- 하나의 웹앱을 만든다.
- 공개 영역과 보호 영역이 명확히 나뉜다.
- 인증 이후에만 접근 가능한 앱 영역이 있다.
- 기능은 앞으로 계속 추가된다.
- 전체 UI 뼈대는 먼저 만들고, 기능은 모듈 단위로 각개격파한다.
- 원격 인프라는 Cloudflare만 사용한다.
- 초기에 편해 보이는 임시 결합보다, 나중에 후회하지 않는 구조를 우선한다.

---

## 1. 이 문서의 목적

이 문서의 목적은 세 가지다.

첫째, 기능이 늘어나도 코드가 서로 엉키지 않게 한다.

둘째, 나중에 스스로 "왜 이렇게 만들었지"라고 후회할 가능성이 높은 구조적 실수를 미리 막는다.

셋째, 새로운 기능을 추가할 때 매번 설계를 처음부터 고민하지 않고, 이미 정한 기준에 따라 일관되게 구현하게 만든다.

즉, 이 문서는 "예쁘게 설계하기 위한 문서"가 아니라, "무너지는 것을 방지하기 위한 문서"다.

---

## 2. 가장 먼저 고정해야 하는 사고방식

이 프로젝트는 페이지 중심으로 커지면 안 된다.
페이지는 진입점일 뿐이고, 실제 코드의 중심은 기능 모듈이어야 한다.

잘못된 방향은 이런 것이다.

- `/app/page1` 폴더 안에 관련 로직, 상태, API, UI를 전부 몰아넣기
- 비슷한 기능이 다른 페이지에도 필요해지면 복사해서 재사용하기
- 점점 페이지별 중복 코드가 늘어나고, 같은 수정이 여러 군데에 퍼지기

올바른 방향은 이렇다.

- 페이지는 라우트 진입점만 가진다.
- 실제 기능은 `features` 아래에 모인다.
- 기능에 필요한 UI, 상태, API 접근, 검증은 해당 기능 모듈 안에 모인다.
- 페이지는 기능 모듈을 배치하고 조합하는 역할만 맡는다.

한 줄 기준:

**페이지는 조립하고, 기능 모듈은 책임진다.**

---

## 3. 구조가 무너지지 않게 하는 최상위 규칙

### 규칙 1. feature는 다른 feature의 내부를 직접 참조하지 않는다.

예를 들어 `features/profile`은 `features/posts/internal/...` 같은 것을 직접 가져오면 안 된다.
기능끼리 직접 파고들기 시작하면, 언젠가 분리도 교체도 어려워진다.

허용되는 방식은 둘 중 하나뿐이다.

- 공용 계층을 통해 공유한다.
- 명시적으로 공개된 인터페이스를 통해 연결한다.

### 규칙 2. 공용 폴더는 작고 엄격해야 한다.

처음부터 뭐든지 `shared`에 넣기 시작하면 결국 `shared`가 쓰레기통이 된다.
공용은 정말 반복 가능성이 높고, 도메인에 덜 묶인 것만 둔다.

### 규칙 3. 인증, 세션, API, 저장소 접근은 반드시 경계층을 거친다.

컴포넌트가 직접 세션을 읽고, 여기저기서 직접 fetch를 호출하고, 로컬 저장소를 직접 만지기 시작하면 구조가 급속히 더러워진다.
그런 작업은 반드시 정해진 계층을 통해서만 하게 해야 한다.

### 규칙 4. UI와 도메인 로직은 섞지 않는다.

버튼 컴포넌트가 세션 구조를 알고 있으면 안 된다.
페이지 레이아웃 컴포넌트가 API 에러를 직접 해석하면 안 된다.
도메인 로직은 기능 모듈 또는 플랫폼 계층에서 처리한다.

### 규칙 5. 전역 상태는 마지막 수단이다.

아직 한 페이지 안에서 끝나는 상태를 전역으로 올리면, 나중에 의존성이 불필요하게 커진다.
전역 상태는 정말 앱 전체가 공유해야 하는 것만 둔다.

## 4. 추천 폴더 구조

아래 구조는 지금 프로젝트의 방향에 맞는 기본형이다.

```text
src/
  app/
    router/
    providers/
    layout/
  pages/
    home/
    app/
  features/
    auth/
    profile/
    dashboard/
    ...
  shared/
    ui/
    lib/
    types/
    styles/
  platform/
    api/
    auth/
    session/
    storage/
    config/
```

이 구조의 의미는 다음과 같다.

### `app/`

앱 전체를 부팅하고 조립하는 계층이다.
라우터, 전역 provider, 앱 셸, 최상위 레이아웃 등을 둔다.
앱을 어떻게 시작하고 감쌀지 정하는 곳이다.

### `pages/`

라우트 진입점 계층이다.
`/`, `/app` 같은 페이지 단위 엔트리만 둔다.
여기에는 기능의 상세 로직을 많이 넣지 않는다.

### `features/`

이 프로젝트의 핵심 계층이다.
실제 기능은 여기에 산다.
예를 들어 프로필, 대시보드, 검색, 알림, 게시물 같은 기능별로 나눈다.
각 기능은 자신의 UI, 상태, 서버 호출, 검증, 예외 처리를 포함할 수 있다.

### `shared/`

도메인에 종속되지 않는 범용 요소만 둔다.
예를 들면 버튼, 카드, 텍스트 컴포넌트, 날짜 포맷 유틸, 공통 타입, 스타일 토큰 등이 여기에 해당한다.

### `platform/`

인증, 세션, API 클라이언트, 로컬 저장소, 환경 설정 같은 인프라 계층이다.
기능들이 직접 외부 환경과 결합하지 않도록 중간에서 경계를 잡는다.

---

## 5. 각 계층에 두면 안 되는 것

### `pages/`에 두면 안 되는 것

- API 직접 호출 로직
- 긴 비즈니스 로직
- 여러 페이지에서 재사용될 UI 구현
- 세션 구조를 직접 파싱하는 코드

페이지는 "구성"만 해야 한다.

### `shared/`에 두면 안 되는 것

- 특정 기능에 종속된 컴포넌트
- 도메인 전용 타입
- 아직 한 번밖에 안 쓰인 코드
- 특정 페이지 전용 레이아웃

`shared`는 재사용 가능성이 높아야 하며, 도메인 냄새가 약해야 한다.

### `platform/`에 두면 안 되는 것

- 화면 컴포넌트
- 기능별 문구나 UI 흐름
- 특정 feature 내부 규칙

`platform`은 외부 시스템과의 접점만 다뤄야 한다.

### `features/`에 두면 안 되는 것

- 앱 전체 라우터 설정
- 전역 레이아웃 구조
- 다른 feature 내부 파일에 대한 직접 참조

feature는 자기 책임만 가져야 한다.

---

## 6. 기능 모듈을 자르는 기준

기능 모듈은 "화면"이 아니라 "업무 의미"를 기준으로 자른다.

좋은 예:

- `auth`
- `profile`
- `dashboard`
- `notifications`
- `settings`

나쁜 예:

- `home-page-components`
- `app-page-section-1`
- `left-panel-ui`

후자는 나중에 라우트 구조가 조금만 바뀌어도 의미가 무너진다.
기능은 페이지보다 오래 살아야 한다.

기능 모듈 하나의 이상적인 범위는 이렇다.

- 해당 기능에 관련된 UI
- 해당 기능의 상태
- 해당 기능의 API 접근
- 해당 기능의 입력 검증
- 해당 기능의 로딩/에러/빈 상태

즉, 기능 하나를 닫을 때는 "화면 일부"가 아니라 "작동하는 단위"로 닫아야 한다.

## 7. 인증과 세션 설계 원칙

인증 관련 코드는 나중에 가장 많이 흔들릴 가능성이 있는 영역이다.
그러므로 초기에 경계를 잘 잡아야 한다.

### 반드시 지켜야 할 원칙

1. UI 컴포넌트가 직접 세션 저장 방식에 의존하지 않는다.
2. 세션 조회는 한 군데의 진입점을 통해서만 이뤄진다.
3. 현재 사용자가 누구인지 아는 방식은 앱 전체에서 일관되어야 한다.
4. 보호 라우트는 "현재 세션이 유효한가"만 판단하고, 세션 저장 세부 구현은 몰라야 한다.

### 추천 인터페이스

실제 구현이 바뀌더라도, 함수 형태는 아래처럼 안정적으로 유지하는 편이 좋다.

- `getSession()`
- `getCurrentUser()`
- `isAuthenticated()`
- `signOut()`

이렇게 해두면 현재처럼 Cloudflare Worker 경계 뒤의 실제 OAuth/session 구조를 쓰더라도, 이후 세션 보강이나 로그아웃 UX 추가 때 UI를 덜 뜯게 된다.

### 하지 말아야 할 것

- 아무 컴포넌트나 직접 localStorage를 읽게 하기
- 보호 라우트 안에서 세션 구조를 직접 파싱하기
- 버튼 클릭 핸들러 안에 인증 시스템 세부 구현을 넣기

## 8. API 접근 설계 원칙

나중에 가장 후회하기 쉬운 것 중 하나가 여기저기 흩어진 `fetch` 호출이다.

### 원칙

1. UI 컴포넌트 안에서 직접 `fetch`를 남발하지 않는다.
2. API 호출은 기능 모듈 내부의 service 함수나 공통 API 계층을 통해 나간다.
3. 인증 헤더, 에러 포맷, 공통 응답 처리 규칙은 가능한 한 한 곳에 모은다.
4. API 응답 타입은 기능 경계에서 정리한다.

### 추천 방향

- 공통 HTTP 클라이언트: `platform/api`
- 기능별 호출: `features/<feature>/services`

예를 들어 프로필 관련 API는 `features/profile/services`에서 관리한다.
직접 페이지나 버튼에서 `fetch('/api/profile')`를 호출하지 않는다.

## 9. 상태 관리 원칙

상태 관리의 핵심은 "모든 것을 전역으로 만들지 않는 것"이다.

### 상태의 3단계

#### 1) 로컬 상태

한 컴포넌트나 한 페이지 안에서만 쓰는 상태다.
예: 입력창 값, 모달 열림 여부, 탭 선택 상태.

#### 2) 기능 내부 상태

한 feature 내부에서만 공유되는 상태다.
예: 프로필 수정 폼 상태, 대시보드 필터 상태.

#### 3) 앱 전역 상태

앱 전체가 공유해야 하는 상태다.
예: 현재 사용자, 세션 정보, 전역 알림.

### 원칙

- 먼저 로컬 상태로 시작한다.
- 여러 컴포넌트가 함께 써야 하면 feature 내부로 올린다.
- 진짜 앱 전체가 알아야 할 때만 전역 상태로 올린다.

### 하지 말아야 할 것

- 아직 한 페이지에서만 쓰는 상태를 전역 store에 넣기
- 도메인 상태와 UI 상태를 같은 전역 store에 뒤섞기

## 10. 공용 UI 설계 원칙

공용 UI는 구조를 단순하게 유지하는 데 도움이 되지만, 너무 빨리 많이 만들면 오히려 복잡해진다.

### 초기에 공용화해도 좋은 것

- 버튼
- 카드
- 페이지 컨테이너
- 섹션 제목 블록
- 빈 상태 UI
- 앱 셸 구조

### 아직 공용화하지 말아야 할 것

- 특정 페이지 전용 섹션
- 랜딩 페이지 문구 카드
- 한 번만 쓰는 복합 조합 컴포넌트

### 판단 기준

아래 질문에 모두 예라고 답할 수 있을 때만 공용화한다.

1. 두 번 이상 반복되는가?
2. 앞으로도 반복될 가능성이 높은가?
3. 도메인에 너무 묶여 있지 않은가?
4. 공용으로 빼도 이름이 자연스러운가?

---

## 11. 스타일 설계 원칙

스타일은 나중에 가장 조용하게 부채가 쌓이는 영역이다.
그러므로 최소한의 기준을 초기에 잡아두는 게 좋다.

### 반드시 먼저 정할 것

- 색상 토큰
- 글자 크기 단계
- spacing 단위
- border radius
- shadow 단계
- 최대 너비 기준

### 추천 방식

공통 값은 CSS 변수 또는 공통 스타일 파일에 둔다.
페이지별 레이아웃 스타일은 해당 페이지 또는 feature에 둔다.

### 하지 말아야 할 것

- 색상값을 컴포넌트마다 하드코딩하기
- spacing 숫자를 제각각 직접 입력하기
- 공용 버튼과 페이지 전용 버튼이 제각각 다른 모양이 되도록 방치하기

---

## 12. 라우트 설계 원칙

이 프로젝트에서는 라우트가 기능의 중심이 아니라, 기능을 보여주는 입구여야 한다.

### 추천 라우트 체계

- `/` : 공개 랜딩
- `/app` : 보호 루트
- `/app/...` : 로그인 후 기능 페이지들
- `/auth/...` : 인증 관련 콜백 또는 흐름

### 원칙

1. 공개와 보호 경계는 초기에 고정한다.
2. 보호 여부 판단은 라우트 경계에서 먼저 한다.
3. 실제 세부 기능은 라우트보다 feature 중심으로 구현한다.

### 하지 말아야 할 것

- 페이지별로 제각각 인증 검사하기
- `/app` 하위 일부에서만 비밀스럽게 인증을 체크하기
- 인증 검사를 여러 컴포넌트에 중복 구현하기

---

## 13. 에러, 로딩, 빈 상태 규칙

이 항목은 초기에 가볍게 넘어가기 쉽지만, 나중에 품질 차이를 만든다.

각 기능 모듈은 가능한 한 아래 4가지를 스스로 가져야 한다.

- 정상 상태
- 로딩 상태
- 에러 상태
- 빈 상태

즉, 기능 모듈이 "작동하는 화면"이라면, 성공한 순간만 생각하면 안 된다.

이걸 미리 규칙으로 적어두지 않으면 나중에 대부분의 기능이 "정상 상태만 있는 반제품"이 된다.

## 14. 기능 완료 기준

완전 모듈화에서 가장 중요한 것 중 하나는 "무엇을 완료라고 부를지"를 미리 정하는 것이다.

기능 하나를 완료했다고 말하려면 최소한 아래가 닫혀야 한다.

1. UI가 있음
2. 상태가 있음
3. 입력 검증이 있음
4. 서버 연결 또는 데이터 흐름이 있음
5. 로딩/에러/빈 상태가 있음
6. 페이지에서 실제로 쓸 수 있음

이 기준이 없으면 "절반 구현된 기능"이 계속 쌓인다.

## 15. 지금 미리 만들면 좋은 문서들

나중에 후회하지 않으려면 아래 문서들을 너무 늦게 만들면 안 된다.

### 1) 구조 원칙 문서

바로 이 문서다.
무엇을 어디에 둘지, 무엇을 금지할지 적는다.

### 2) 진행 기록 문서

현재 이미 만들기 시작한 progress-memory 문서다.
각 단계에서 뭘 했고 왜 했는지 남긴다.

### 3) 기능 등록 문서

앞으로 추가할 기능 목록을 적고, 각 기능이 어느 feature로 들어갈지 정한다.

### 4) 공용 컴포넌트 기준 문서

언제 shared로 올리고, 언제 feature 내부에 둘지 기준을 적는다.

### 5) 인증/세션 규칙 문서

현재 사용자와 보호 라우트가 어떤 규칙으로 동작하는지 적는다.

---

## 16. 초기에 흔히 하는 실수와 방지 규칙

### 실수 1. shared를 너무 크게 만들기

방지 규칙: 두 번 이상 반복되기 전에는 shared로 올리지 않는다.

### 실수 2. page 안에 모든 것을 넣기

방지 규칙: page는 조립만 하고, 로직은 feature로 보낸다.

### 실수 3. fetch를 아무 데서나 호출하기

방지 규칙: API 호출은 기능 service 또는 platform api를 통해서만 한다.

### 실수 4. 세션을 여기저기서 직접 읽기

방지 규칙: 세션 접근은 auth/session 계층으로 고정한다.

### 실수 5. 전역 상태 남용

방지 규칙: 로컬 → feature → 전역 순서로만 승격한다.

### 실수 6. "나중에 정리하지 뭐"라고 미루기

방지 규칙: 구조를 깨는 예외는 문서 없이 만들지 않는다.

## 17. 앞으로 새 기능을 추가할 때의 체크리스트

새 기능을 만들 때마다 아래 질문을 먼저 본다.

1. 이 기능은 어느 feature에 속하는가?
2. 이 기능은 page가 아니라 feature 안에 들어가야 하는가?
3. 공통 UI가 필요한가, 아니면 feature 전용 UI인가?
4. API 호출은 어디에 둘 것인가?
5. 상태는 로컬/feature/전역 중 어디에 둘 것인가?
6. 로딩/에러/빈 상태는 준비되어 있는가?
7. 다른 feature에 직접 의존하지 않는가?
8. 완료 기준을 만족하는가?

이 질문을 먼저 거치면 대개 후회할 코드를 훨씬 덜 만들게 된다.

## 18. 지금 시점의 권장 실행 순서

현재 프로젝트 상황에서 지금 당장 하면 좋은 것은 다음과 같다.

1. `src` 폴더 구조를 위 원칙에 맞게 천천히 정리한다.
2. `app`, `pages`, `features`, `shared`, `platform`의 역할을 고정한다.
3. 3단계 UI를 만들 때부터 `PageContainer`, `AuthButton`, `AppShell` 정도만 공용화한다.
4. `auth.ts` 같은 경계 모듈은 이미 붙은 실제 Worker 기반 auth/session 흐름을 감춘 채, 이후 단계 변화에도 인터페이스를 안정적으로 유지한다.
5. 다음 기능부터는 page 안이 아니라 feature 안에 실질 구현을 넣는다.

## 19. 마지막 원칙

이 프로젝트에서 가장 중요한 문장은 이것이다.

**빨리 만드는 것보다, 나중에 덜 후회하는 구조로 만드는 것이 더 중요하다.**

당장 조금 번거로워 보여도,

- 경계를 먼저 정하고
- 기능을 feature 단위로 닫고
- 공용화는 신중하게 하고
- 인증/세션/API는 반드시 경계층을 거치게 만들면

나중에 기능이 늘어날수록 오히려 속도가 붙는다.

반대로,

- 페이지 안에 다 넣고
- shared를 아무거나 담는 폴더로 만들고
- fetch와 세션 접근을 여기저기 흩뿌리면

초반에는 빨라 보여도 결국 스스로 그 구조를 두려워하게 된다.

이 문서는 그 상태를 막기 위한 기준이다.

## 20. 구현 후 확인 사항

구현이 끝난 뒤에는 아래 항목을 최소한 확인한다.

- 인증 상태를 boolean 하나로 단순화하지 않았는지 확인한다.
- 보호 라우트가 `loading`, `authenticated`, `unauthenticated`를 구분하는지 확인한다.
- 인증 실패 리다이렉트 시 원래 목적지 정보(`pathname`, `search`, `hash`)를 보존하는지 확인한다.
- 세션 경계와 UI 로직이 섞이지 않았는지 확인한다.
- 현재 단계 범위를 넘는 구현(내구성 세션 저장소, 계정 메뉴 확장, 보호 앱 기능 대형 확장)이 들어가지 않았는지 확인한다.
- `npm run format`, `npm run lint`, `npm run typecheck`, `npm run build`를 통과했는지 확인한다.
